# 世界盃 2026 — 伺服器自動部署說明

讓你的伺服器自動從 GitHub 拉最新 `data/` 並重啟容器。提供兩種方式,**擇一**即可（也可兩者並用,timer 當作 webhook 的保險）。

前置：repo 已 clone 到伺服器,Docker 容器已用 `docker compose up -d` 跑起來。以下假設 repo 在 `/opt/tonic-fifa-wc2026`,若不同請改各檔案中的 `REPO_DIR` / 路徑。

```bash
# 首次安裝（在伺服器上）
sudo git clone https://github.com/tonylnng/tonic-fifa-wc2026.git /opt/tonic-fifa-wc2026
cd /opt/tonic-fifa-wc2026
# 改 docker-compose.yml 裡的 SITE_PASSWORD 後啟動
docker compose up -d
chmod +x deploy/update.sh deploy/webhook.py
```

核心更新腳本是 `deploy/update.sh`：fetch → 比對版本 → 有變更才 `git reset --hard`,並判斷只動到 `site/data/` 就 `restart`、動到程式碼就 `up -d --build`。兩種方案都呼叫它。

---

## 方案 A：Webhook 即時觸發（推薦,秒級更新）

排程每輪 push 後,GitHub 立即通知你的伺服器。需要伺服器有一個對外可達的埠（或用 Tailscale Funnel / Cloudflare Tunnel 對外）。

### 1. 安裝 webhook 監聽服務

```bash
# 產生一組密鑰並記下來（GitHub 那邊要填一樣的）
openssl rand -hex 20

# 編輯 service 檔,把 WEBHOOK_SECRET 改成上面產生的密鑰
sudo cp deploy/wc2026-webhook.service /etc/systemd/system/
sudo nano /etc/systemd/system/wc2026-webhook.service   # 改 WEBHOOK_SECRET

sudo systemctl daemon-reload
sudo systemctl enable --now wc2026-webhook.service
sudo systemctl status wc2026-webhook.service
```

監聽器預設在 `0.0.0.0:9000/webhook`,驗證 GitHub 的 HMAC-SHA256 簽章後背景執行 `update.sh`。

### 2. 對外暴露（三選一）

- **Nginx/Caddy 反向代理**：把 `https://你的網域/gh-webhook` 代理到 `localhost:9000/webhook`。
- **Tailscale Funnel**：`tailscale funnel 9000`（最省事,免自己處理憑證）。
- **Cloudflare Tunnel**：`cloudflared tunnel --url http://localhost:9000`。

### 3. 在 GitHub 設定 Webhook

到 repo → Settings → Webhooks → Add webhook：

- **Payload URL**：你對外的網址（例如 `https://你的網域/gh-webhook`）
- **Content type**：`application/json`
- **Secret**：步驟 1 產生的密鑰
- **Which events**：Just the push event
- 按 Add webhook,GitHub 會送一個 ping,監聽器回 `pong` 即成功。

之後排程每次 push,伺服器數秒內自動更新。

## 方案 B：systemd timer 定時輪詢（不需對外開埠）

伺服器每 5 分鐘自己去 GitHub 檢查一次,有更新才動作。最單純,適合不想開對外埠的環境。

```bash
sudo cp deploy/wc2026-deploy.service /etc/systemd/system/
sudo cp deploy/wc2026-deploy.timer   /etc/systemd/system/
# 如路徑非 /opt/tonic-fifa-wc2026,改 .service 內的 Environment=REPO_DIR=...
sudo nano /etc/systemd/system/wc2026-deploy.service

sudo systemctl daemon-reload
sudo systemctl enable --now wc2026-deploy.timer

# 確認下次執行時間
systemctl list-timers wc2026-deploy.timer
# 手動跑一次測試
sudo systemctl start wc2026-deploy.service
```

輪詢間隔想改：編輯 `wc2026-deploy.timer` 的 `OnUnitActiveSec=5min`。

> 因為排程是每 8 小時（16 強起每 4 小時）才更新一次,輪詢間隔設 5–15 分鐘已綽綽有餘,且 `git fetch` 無變更時幾乎零成本。

---

## 用 cron 取代 systemd（如果你偏好 cron）

不想用 systemd 的話,方案 B 也可以直接用 crontab：

```bash
# crontab -e
*/5 * * * * REPO_DIR=/opt/tonic-fifa-wc2026 /opt/tonic-fifa-wc2026/deploy/update.sh
```

## 設定速查

`update.sh` 可用環境變數覆寫：

| 變數 | 說明 | 預設 |
|------|------|------|
| `REPO_DIR` | repo 在伺服器路徑 | `/opt/tonic-fifa-wc2026` |
| `BRANCH` | 追蹤分支 | `master` |
| `COMPOSE` | compose 指令 | `docker compose`（舊版改 `docker-compose`） |
| `FORCE_REBUILD` | 設 `1` 則每次強制重新 build | `0` |
| `LOG` | 日誌路徑 | `/var/log/wc2026-deploy.log` |

## 驗證與排錯

```bash
# 看更新日誌
tail -f /var/log/wc2026-deploy.log
# 看 webhook 服務日誌（方案 A）
sudo journalctl -u wc2026-webhook.service -f
# 看 timer 執行紀錄（方案 B）
sudo journalctl -u wc2026-deploy.service -f
# 確認容器有起來
docker compose ps
```

## 運作邏輯小結

- 雲端排程產生新預測 → push 到 GitHub。
- 你的伺服器（webhook 即時／timer 輪詢）偵測到新 commit → `git reset --hard` 拉最新。
- 只動到 `site/data/`：`docker compose restart`（快）。動到程式碼：`docker compose up -d --build`。
- 因 `docker-compose.yml` 預設把 `./site/data` 以 volume 掛載進容器,資料更新後 restart 即生效。
